# Mentoring and enabling others

**Parent:** [`README.md`](../README.md)

## What this probes

Helping **others** ship and ramp: pairing, clear expectations, reviews as teaching—not “I manage a team.” As an IC, this is often **spot mentoring** and **onboarding** quality.

## Example questions

| # | Question | Note |
| --- | --- | --- |
| 1 | Tell me about helping a new hire ramp. | Concrete cadence. |
| 2 | Pair programming or mobbing—what outcome did you drive? | If rare, narrow scope. |
| 3 | When did you unblock a teammate technically? | Technical + interpersonal. |
| 4 | Tell me about knowledge sharing that scaled beyond one person. | Doc, talk, tool. |
| 5 | How do you use code review as mentoring? | Specific review example. |
| 6 | When did you give constructive feedback that helped someone improve? | Boundary if none. |
| 7 | Tell me about an onboarding checklist or doc you created or improved. | Pick one Beyond ramp (C++, Python pipeline, or FE) below. |
| 8 | When did you hand off ownership cleanly? | Transition plan. |
| 9 | When did you set expectations with a struggling peer (without being their manager)? | Tact + escalation awareness. |
| 10 | Tell me about teaching a concept that finally clicked for someone. | Technique you used. |
| 11 | How do you help others debug without taking over their keyboard forever? | Short pairing story. |
| 12 | When did you set expectations for a new owner? | Handoff + docs. |

## Aligned stories

- [Clojure SDK, calendar limits, and a mentoring handoff](../work-experience/2022-orkes-sdk.md#clojure-sdk-calendar-limits-and-a-mentoring-handoff) — **Handoff / pairing / “reset with manager”:** structured context transfer to a teammate when a **sixth-language** track plus calendar overload made solo ownership unrealistic; **non-overlap** with Beyond: customer SDKs and **remote** checkpoints, not trading-domain ramp. Strong for **clean ownership transfer**, **expectations with manager**, and **debugging help without heroics** prompts.
- [Onboarding a Python engineer into market-data and strategy-evaluation](../work-experience/2019-beyond-hft.md#onboarding-a-python-engineer-into-market-data-and-strategy-evaluation) — **Pipeline / “data science side” ramp:** Python-first contribution to **ingest → ETL → consumer / evaluator** context and replay-driven decisions; **non-overlap** with C++ story (different surface and hire profile). Strong for **knowledge sharing beyond one person**, **short pairing**, and **handoff quality** where the lever was **bounded tasks + review-as-teaching**.
- [Navigating front-end ownership churn and an async replay handoff](../work-experience/2019-beyond-hft.md#navigating-front-end-ownership-churn-and-an-async-replay-handoff) — **FE + API contracts + local iteration:** explicit **WebSocket/HTTP** boundaries, rotation briefs, and **build/run/deploy** orientation so new UI owners were not blocked on tacit env knowledge; **non-overlap** with pipeline story (UI seat vs. Lambda/consumer path). Strong for **unblock a teammate**, **short pairing**, and **handoff quality**.

## Gaps and detail drills

- **Thin risk:** easy to blur into “we wrote docs.” **Drill:** **who** you paired with (role level), **how many** sessions or weeks, **what they shipped after**, one **outcome** metric if any.
- For PIP / formal performance prompts, see [sensitive-prompts](sensitive-prompts.md)—many ICs have **no** story; boundary + redirect is OK.
